# Запросы для сильной модели по развитию Docmancer

Этот файл содержит набор отдельных prompt-ов для сильной модели. Их лучше отправлять **не все сразу**, а по одному: сначала получить глубокий ответ по одному направлению, затем использовать его как вход для следующего.

## Что прикладывать к каждому запросу

Ты правильно понял: к запросам желательно докидывать исходный контекст.

Минимальный пакет контекста:

1. `DOCMANCER_PRODUCT_BRIEF.md` — базовая выжимка по Docmancer.
2. Аналитический roadmap, который уже дала сильная модель.
3. Текущий prompt из этого файла.

Для последующих запросов дополнительно прикладывай результаты предыдущих ответов сильной модели, но коротко:

- полный ответ — если модель может переварить большой контекст;
- или 10–20 bullet points с решениями, если контекст ограничен.

Рекомендуемый порядок:

1. `01_agent_proof_mcp_docs_ux.md`
2. `02_registry_source_identity.md`
3. `03_product_positioning.md`
4. `04_project_aware_version_resolution.md`
5. `05_retrieval_quality_eval.md`
6. `06_first_run_dx_doctor.md`
7. `07_merged_execution_roadmap.md`

Почему так: сначала закрываем самый конкретный и опасный UX-разрыв (`needs_docs_url` и уход агента в WebFetch), затем нормализуем registry/source model, затем уже уточняем стратегию, ecosystem coverage, retrieval quality и DX.

## Как просить модель отвечать

В каждом prompt-е проси модель:

- не пересказывать общий roadmap;
- не давать только high-level советы;
- давать конкретные decisions, acceptance criteria, тесты и implementation plan;
- явно отделять `must-have`, `should-have`, `could-have`;
- отмечать риски и trade-offs;
- формулировать результат так, чтобы его можно было превратить в GitHub issues / PR plan.
