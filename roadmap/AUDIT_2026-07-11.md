# DocAtlas roadmap audit — 2026-07-11

## Audit baseline

- Repository state: `main` at `0664fcf` after PR 38.
- Scope: roadmap 01–09, primary Docs MCP workflow, external-library ingest, project-doc workflow, evaluation, tests, CI, packaging, installer, and active user/model documentation.
- This is a code-and-evidence audit. A merged PR was not counted as completion when its acceptance evidence was absent.

## Executive conclusion

The roadmap is not finished. Tasks 02 and 07 are complete for their deliberately narrow scope, and task 01 completed its named Node extraction. Tasks 03–06 and 09 are partial. Task 08 is open.

The product direction is sensible: a local-first, source-attributed project documentation layer is useful, and exact dependency context can complement it. The current implementation is not yet a credible Context7 replacement. The strongest evidence is the repository's own benchmark: the decisive task-level run is marked `NEGATIVE_SIGNAL`, all four conditions resolved 0/3 tasks, and the DocAtlas lanes used more time and input context without a correctness gain.

## P0 findings

### The documented install does not install the documented product

- `docmancer/_version.py` is still `1.1.1`.
- Tag `v1.1.1` is more than 100 commits and 180 tracked paths behind the audited `main`.
- `scripts/install.sh` installs the latest PyPI `doc-atlas`, currently `1.1.1`.
- That release predates the current `prepare_docs` and `docs_status` public workflow.

The first-screen README workflow therefore cannot be obtained through its own advertised one-line install. Task 15 adds an artifact-level gate; task 08 must make the current limitation explicit immediately.

### External ingest still violates task 09

- `prepare_docs(prefetch_library_docs)` does not propagate singular `version` through the public dispatcher; an exact request can become `latest`.
- The controller marks a timed-out job as cancelling but continues waiting for a hung worker, so the overall deadline is not hard.
- Jobs use in-memory state and unbounded daemon threads; there is no admission control or restart recovery.
- Real fetch exceptions are often flattened to `None`, `ValueError`, or a later extraction error, so VPN/proxy failures are not classified reliably.
- Missing library guidance can send the model back to `get_docs_context(allow_network=true)` instead of the explicit `prepare_docs` lifecycle.
- A multi-page refresh can still discard usable staging when another page raises, so the existing `partial` status does not reliably preserve/report partial page success.
- No committed Kotlin live-smoke result proves the incident workflow succeeds.

Tasks 11–14 and 30–31 split the unfinished umbrella task into reviewable PRs.

### The fetch boundary is weaker than the documentation claims

- URL validation blocks literal private IPs but does not establish a complete DNS/redirect policy.
- Redirects can be followed without re-validating the final destination against private-network and source policy.
- URL userinfo, response-size/decompression bounds, and centralized secret redaction are incomplete.
- Raw exception text can reach registry or job state.

Task 12 owns this boundary. Do not expose arbitrary remote acquisition more broadly before it is complete.

### Exact provenance is incorrectly conflated with instruction trust

The current trust contract can label exact dependency documentation as `trusted`, even though source/version provenance does not make page text safe agent instructions. The threat model also treats documentation prompt injection as a non-goal. Task 28 separates provenance, repository authority, exactness, and instruction trust before broader crawled content is fed to agents.

### A curated source is deterministically invalid

The `flutter_bloc` source expands to `pub.dev`, while its allowlist contains only `bloclibrary.dev`. Target validation rejects the generated URL. Existing tests cover `go_router`, not every manifest entry. Task 10 is the smallest safe next code fix after the documentation truth pass.

## Evaluation findings

### Context7 parity is scaffolding, not a comparison

- `eval/context7_parity/dataset.jsonl` contains 150 rows.
- `run_docatlas.py` re-scores a supplied trace; it does not execute DocAtlas or capture results.
- No complete sanitized result/report is committed.
- Relevance can pass on the right host plus symbol even when the path and section are wrong.
- An existing result object can count as snippet presence even when its snippet is empty.
- Current coverage rules cannot represent a complete cold and warm pair for every case.
- Many questions are templated variants of one topic and several exact requests point at rolling/latest corpora.

Task 18 replaces claim-oriented scoring with a paired, auditable capture protocol.

### Cold start is mostly unproven

- The manifest has 30 entries, but only three are exact-version sources.
- The other versioned parity requests correctly decline silent fallback, so they receive no curated cold-start benefit.
- `canonical_source_identity` is recorded as metadata but is not a demonstrated identity-plus-content-hash snapshot cache.
- Preferred seeds are retained but not used by target construction.
- There is no measured under-10-second cold SLO or zero-network warm proof.
- Workspace and duplicate-version resolution remains incomplete.

Tasks 16 and 17 finish source and dependency identity separately.

### Product proof is currently negative

The decisive benchmark report honestly records `NEGATIVE_SIGNAL`: no lane resolved any of the three tasks, while the DocAtlas tool-recommended lane had higher median duration and much larger context volume than repo-only. The three newer candidates are correctly marked too easy and non-differentiating.

This does not prove the product has no value. It proves that retrieval metrics and tool adoption are insufficient. Task 23 requires independent real-project tasks and an explicit improve-or-pivot decision rule.

## Local project-documentation findings

### Missing-doc evidence can be falsely complete

`evidence_complete` is derived from whether any evidence was found, rather than whether every required section has enough evidence. A minimal project can therefore tell a weaker model to inspect only a manifest while still asking it to describe entrypoints, modules, runtime flow, and tests. Task 19 adds per-section gaps and configurable documentation roots.

### Section metadata is written but not consumed

Indexing stores `project_doc_sections`, but production impact analysis reparses Markdown and does not use the stored hash/section metadata. CI supplies changed paths, while changed symbols normally require manual input. Task 20 closes that loop.

### Completed roadmap prompts can contaminate retrieval

The project docs scanner discovers `roadmap/` as supporting documentation. Before this audit, the indexable roadmap files said task 01 was merely `Started`, omitted task 09 from execution order, and did not distinguish completed from active prompts. A sync could therefore make a model recommend already merged work. The README status table is now the authority; task 19 must encode authority/status in retrieval rather than relying only on prose.

## Tests, CI, and release findings

- Under the ambient SOCKS proxy, the full suite fails while constructing environment-aware HTTP clients when the optional SOCKS transport is absent.
- Without proxy variables, multiple Packs tests still make real DNS/preflight calls to `example.com`; the suite is not hermetic.
- CI currently tests Python 3.11 and 3.12 while package metadata claims 3.13.
- The publish workflow tests editable source, not the exact wheel installed into a clean environment.
- The existing stdio smoke targets advanced Packs, not the primary Docs MCP flow.
- `Production/Stable` is premature until the built artifact passes the primary-flow smoke.

Tasks 30 and 15 own these issues.

## Documentation and model-guidance drift

- `docs/DOCMANCER_PRODUCT_BRIEF.md` still leads with Project Patch Contract Runtime.
- `CHANGELOG.md` does not summarize the recent Docs MCP, dependency, impact, bootstrap, and ingest work.
- `.github/copilot-instructions.md` contains an absolute developer path and a legacy CLI-first flow.
- Wiki command and troubleshooting pages expose old direct tools or the wrong package name.
- The hero installer registers MCP but does not install the compact project bootstrap from task 02.
- Root `doc-atlas --help` hides the `mcp` command group even though onboarding shows `doc-atlas mcp docs-serve`.
- `.gitignore` ignores `docs/*` and whitelists only selected files; several tracked active docs are outside that allowlist, so a new neighboring document can be silently ignored.
- Active detailed docs are heavily duplicated: `capabilities.md`, the product brief, and the MCP guide alone exceed 3,000 lines, working against the weaker-model/token-budget goal.

Task 08 fixes factual drift. Task 21 then tests whether a weaker model actually chooses the correct public tool from real installed guidance.

## Architecture and operational cost

The bounded Node extraction was useful, but major application modules remain roughly 1,000–1,700 lines. Background jobs are implemented in more than one service, and broad exception handling remains common. The default installation also includes vector and multi-format document dependencies that are not required for every lexical local-doc workflow.

Tasks 24–27 and 29 address install and change-cost budgets only after the primary behavior is stable.

## Decision

Continue the product, but narrow the near-term claim:

> DocAtlas is a local-first, source-attributed documentation context layer for coding agents. External exact-version library context is under active validation; Context7 parity is not yet established.

Execute Stage A first, including tasks 28, 30, and 31 plus the separately owned live/post-publish gates. Then finish the local-project loop and run task 23's decision benchmark. Continue with tasks 16–18 only if that predeclared decision supports further external-library parity investment; otherwise keep DocAtlas focused on local project documentation and integrate Context7 as a complementary external source.
